# Low-Coverage Official Product Backfill Design

## Background

The OCR Insurance knowledge library now has 37,531 local `knowledge_records`. The previous responsibility-repair pass reduced blank invalid responsibility rows to zero: every local row has non-empty `pageText`.

The next useful expansion is not another blank-responsibility repair. It is a product-count backfill for insurers with low local coverage, using the same official-source and Feishu-sync rules as prior crawls.

## Goal

Add new official insurer product-material records for low-coverage companies.

Initial targets are low-count or suspiciously split company names such as:

- `中国人民人寿保险股份有限公司`
- `英大人寿`
- `海保人寿`
- `华汇人寿`
- `小康人寿`

The exact first batch should include three to five companies after source probing. Success means each inserted row has an official source URL, a non-empty insurance responsibility excerpt, a traceable parser/source note, a local SQLite ID, and a matching Feishu row after read-back verification.

## Non-Goals

- Do not repair blank responsibility rows; the current blank-invalid count is zero.
- Do not use third-party article, broker, forum, encyclopedia, or review text as responsibility content.
- Do not ingest company introductions, marketing-only pages, catalogs without material links, rate tables, cash-value tables, or benefit illustrations as responsibility text.
- Do not rewrite unrelated existing rows or normalize all historical company names in this batch.
- Do not modify production SQLite, production runtime, secrets, or deployment config.
- Do not add broad new abstractions unless a target insurer requires a small adapter following existing crawler patterns.

## Source Rules

Allowed sources:

- insurer official websites
- official public-disclosure pages
- official product catalogs
- official terms PDFs
- official product manuals when they include concrete responsibility text
- official PDF/ZIP attachment hosts that are part of the insurer disclosure flow

Disallowed sources:

- third-party content sites
- sales articles
- summaries generated by non-official pages
- product descriptions without official material links
- OCR/model guesses when no official material is available

Terms documents are preferred over product manuals. Product manuals may be inserted only when they contain clear triggers and payout rules.

## Responsibility Acceptance Rules

An excerpt may be inserted only when it contains real insurer obligation text. It must include at least one concrete trigger and payout or reimbursement rule, such as death, disability, disease, medical expense, annuity, survival benefit, maturity benefit, birthday or longevity benefit, premium waiver, or similar savings-style responsibility.

Reject excerpts when they are:

- table of contents only
- responsibility-exclusion sections
- claim application procedures
- dividend, bonus, cash-value, surrender, policy-loan, investment-account, or benefit-illustration text only
- continuation fragments that start mid-clause and depend on missing earlier items
- OCR garbage or source text with unreadable key terms

If a source is official but the responsibility section cannot be extracted reliably, record it in the run report as skipped. Do not insert an empty or low-confidence row to inflate counts.

## Recommended Approach

Use a low-coverage company-first batch.

This is preferred over a large-company sweep because low-count companies are more likely to produce net-new records. The batch should still check company aliases before writing so that full legal names do not create duplicate Feishu tables when a short existing table name is correct.

## Data Flow

1. Load current local SQLite coverage from `.runtime/local/policy-ocr.sqlite`.
2. Select three to five low-coverage target companies.
3. For each target, identify official disclosure/catalog/material entry points.
4. Confirm official homepage and material attachment domains.
5. Collect official product-material metadata.
6. Normalize stable material URLs and compare them against local `knowledge_records.url`.
7. Download only new official PDF/ZIP material.
8. Extract the focused insurance responsibility section using text extraction first, then local Vision OCR when text extraction fails.
9. Classify responsibility quality.
10. Build a timestamped insert plan.
11. Back up SQLite.
12. Insert only eligible new records through the existing knowledge-state path.
13. Sync each company to its Feishu Base table with dry-run, write, and read-back verification.
14. Produce a final report with local and Feishu counts.

## Insert Rules

Automatic insert is allowed only when all of these are true:

- company has been mapped to the correct canonical table name for this project
- material source is official
- normalized material URL is not already represented locally
- product name and material title are non-empty
- material type is `terms` or a qualifying `product_manual`
- `pageText` is non-empty
- responsibility excerpt passes the acceptance rules
- parser/source fields identify how the text was extracted

Records must be skipped when:

- source is unofficial or cannot be verified
- official material cannot be downloaded
- PDF/ZIP is unreadable after text extraction and local OCR attempts
- extracted text is empty or not responsibility text
- only partial/truncated text is available and cannot be bounded to a complete responsibility section
- an anti-bot or human-verification page blocks reliable collection

## Company Alias Handling

Before writing, compare target company names with existing local and Feishu names.

Examples to review:

- `中国人民人寿保险股份有限公司` may belong under the existing `人保寿险` table.
- `阳光人寿保险股份有限公司` may belong under the existing `阳光人寿` table.

Alias handling is conservative. If the source company clearly maps to an existing company table, write to that table. If mapping is uncertain, stop and report the ambiguity instead of creating a duplicate Feishu table.

## SQLite Write

Use the active development database:

`.runtime/local/policy-ocr.sqlite`

Before each write batch, create a SQLite backup under:

`.runtime/local/backups/`

Inserted rows should preserve:

- `company`
- `productName`
- `productType` when available
- `salesStatus` when available
- `title`
- `url`
- `sourceType`
- `materialType`
- `official: true`
- `evidenceLevel`
- `officialDomain`
- `parser`
- `pageText`
- `qualityStatus`
- `qualityReason`
- timestamps
- PDF archive metadata when available

Do not manually edit production data. Do not rewrite unrelated companies.

## Feishu Sync

Sync one insurer to one existing Feishu table whenever possible.

Each sync must:

- run dry-run before write
- sync only newly inserted local IDs
- skip existing remote local IDs
- write to the canonical company table
- read back rows by `本地ID`, URL, or stable product/title key
- require expected count equals found count
- require remote responsibility text to be non-empty

If Feishu write or read-back verification fails, keep the local insert report and stop before claiming the batch is complete.

## Artifacts

Use timestamped artifacts under `.runtime/`, including:

- target/company selection report
- official source probe report
- collected catalog/material metadata
- extraction report
- insert plan
- SQLite backup path
- insert report
- Feishu dry-run report
- Feishu write report
- Feishu read-back verification report

The final report should include per-company counts for:

- official material candidates
- already represented URLs
- downloaded PDFs/ZIPs
- valid responsibility records
- inserted SQLite rows
- synced Feishu rows
- skipped records by reason
- blocked sources by reason

## Error Handling

If an official site returns human verification, slider verification, login walls, request congestion, or a non-material HTML page instead of a PDF, stop that company and report the exact URL and blocker.

If a company uses JavaScript-rendered catalogs, inspect the rendered page and network calls before deciding whether a crawler adapter is needed.

If a material host is not in the existing official-domain profile but is clearly an official attachment host linked from the insurer disclosure flow, record the evidence in the run report before writing.

If one company succeeds and another is blocked, sync and verify the successful company only if its write batch is complete and independently verifiable.

## Verification

Before reporting completion, verify:

- active database path is `.runtime/local/policy-ocr.sqlite`
- SQLite backup exists
- local count increased by exactly the inserted count
- every inserted row has non-empty `pageText`
- inserted rows do not include invalid-empty responsibility text
- inserted rows have official URLs and parser/source notes
- duplicate normalized URLs were not inserted
- Feishu dry-run matched expected new rows
- Feishu write succeeded
- Feishu read-back found all expected rows
- final report separates inserted, already represented, skipped, blocked, and unresolved records

No full test suite is required for a data-only batch. If crawler code changes, run syntax checks and the nearest focused crawler tests before using the changed crawler for writes.
