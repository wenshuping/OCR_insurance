# 保险责任指标质量治理设计

> 日期: 2026-06-20
> 状态: 用户已确认设计
> 范围: 全险种指标审计框架；第一期仅允许年金/两全/养老现金流高置信候选写入开发库

## 目标

建立一套可分险种推进的责任指标质量治理流程。第一期不只为年金险写临时规则，而是建设统一的审计、候选修复、写库 gate 和验证机制。

第一期目标:

- 全险种审计: 年金、医疗、重疾、意外、身故/寿险、豁免都产出问题清单。
- 年金现金流修复: 仅 `cashflow_annuity` lane 的高置信候选允许写入开发库。
- 其他险种暂不写库: 医疗、重疾、意外、身故/寿险、豁免只输出 dry-run 候选和阻塞原因。
- 保护现金流质量: 保险公司确定返钱的责任进入现金流，但疾病、身故、全残、医疗费用、豁免不能被误当普通返钱。

## 范围外

- 不直接修改生产库。
- 不重写 OCR 或官方资料抓取流程。
- 不把医疗、重疾、意外、身故/寿险、豁免候选在第一期写入数据库。
- 不用 OCR-only 文本替代官方知识库证据。
- 不自动确认可选责任已经投保。
- 不把审计报告当作 durable source of truth；持久化数据仍以 SQLite 业务表为准。

## 当前问题

开发库中已存在大量 `insurance_indicator_records`，但质量问题集中在以下几类:

- 责任名泛化，例如 `教育/养老金/两全等返还` 不能表达真实责任。
- 计算基数泛化，例如 `basis=条款载明`，导致现金流引擎无法知道是保费、基本保额、基本责任保险金额，还是可选责任保险金额。
- 同一条款中的多个责任未拆分，例如 `关爱年金`、`生存保险金`、`祝寿金` 只落成一条泛化现金流。
- 疾病关爱金、恶性肿瘤关爱金被误归为现金流。
- 医疗险中免赔额或限额被误当成保险金公式。
- 多个不同给付比例缺少 condition，后续无法区分社保/非社保、计划档位或年龄阶段。

## 设计原则

1. 责任名、触发条件、给付口径一起判断，不能只靠关键词。
2. 全险种共用审计与 dry-run 框架，避免每个险种重复写 DB 读写和安全逻辑。
3. 第一期写库只开放年金现金流 lane，其他 lane 只报告。
4. 每个候选修复必须带官方证据和置信度。
5. 可选责任默认可见但不计入计算，除非保单确认选择且指标已量化。
6. 写库前备份开发 SQLite，写后 rerun dry-run，确认本批高置信候选收敛。

## 架构

新增或重构为一个统一的指标治理入口，内部按 lane 分发。

```text
knowledge_records + insurance_indicator_records
  -> audit runner
  -> lane audits
  -> candidate builder
  -> write gate
  -> report or SQLite write
  -> post-write dry-run and affected policy refresh marking
```

### Audit 层

Audit 层只读扫描开发库，产出问题清单。

每条 issue 包含:

- `lane`
- `issueType`
- `severity`
- `company`
- `productName`
- `sourceRecordId`
- `currentIndicators`
- `sourceExcerpt`
- `reason`

### Candidate 层

Candidate 层把 issue 转成可审查的建议修复，不直接写库。

每个 candidate 包含:

- `lane`
- `operation`: `insert`, `update`, `replace`, `mark_pending_review`, `report_only`
- `confidence`: `high`, `medium`, `low`
- `writeAllowed`
- `blockedReason`
- `sourceRecordId`
- `currentIndicator`
- `proposedIndicator`
- `sourceExcerpt`
- `reason`

### Apply 层

Apply 层第一期只允许 `cashflow_annuity` 的高置信候选写入。

写入条件:

- `lane = cashflow_annuity`
- `confidence = high`
- 使用官方知识库 source record 作为证据。
- 触发条件是生存、到龄、满期、保单周年日、保证领取期等确定返钱条件。
- `value`, `unit`, `basis`, `formulaText` 能明确识别。
- 不涉及疾病、身故、全残、医疗费用、豁免。
- 可选责任若未确认选择，不进入计算，只保留 optional scope 和候选状态。

## Lane 划分

### cashflow_annuity

覆盖年金、两全、养老、教育、满期、祝寿、生存返还等确定返钱责任。

### medical_formula

覆盖医疗费用、免赔额、给付比例、社保/非社保、药品/器械/门诊/住院公式。

### critical_illness

覆盖重大疾病、中症、轻症、恶性肿瘤、特定疾病、疾病关爱金、额外给付、多次给付。

### accident

覆盖意外身故、意外全残、意外伤残、交通意外倍数、意外医疗。

### death_life

覆盖身故、全残、高残、现金价值、已交保费、基本保额、年龄分段、max/sum 公式。

### waiver

覆盖投保人或被保险人豁免后续保险费。

## 年金现金流责任识别

年金现金流不能只识别少数固定词。第一期应把责任名候选和触发条件确认分成两步。

### 责任名候选

年金类:

- `年金`
- `养老年金`
- `养老保险金`
- `后续年金`
- `保证领取年金`
- `保证给付年金`

生存类:

- `生存保险金`
- `特别生存保险金`
- `生存金`

满期类:

- `满期保险金`
- `满期生存保险金`
- `期满保险金`
- `满期金`
- `满期返还`

祝寿类:

- `祝寿金`
- `祝寿保险金`
- `贺寿金`
- `贺岁金`
- `长寿金`

关爱类:

- `关爱年金`

教育成长类:

- `教育金`
- `少儿教育金`
- `高中教育金`
- `大学教育金`
- `深造金`

人生阶段类:

- `婚嫁金`
- `立业金`
- `创业金`

返还类:

- `返还保险费`
- `返还已交保险费`
- `生存返还`

保证领取类:

- `保证领取保险金`
- `保证领取年金`
- `保证给付年金`

保证领取类需要额外判断。有些是年金领取期内继续给付，有些是身故后的差额给付。第一期仅在触发条件和计算口径明确时写入，否则标记 `pending_review`。

### 触发条件确认

责任名命中只生成候选。允许写入现金流还必须命中以下触发条件之一:

- 被保险人生存。
- 犹豫期结束次日生存。
- 每个保单周年日或保单生效对应日生存。
- 到达约定年龄或年龄后的首个保单周年日生存。
- 保险期间届满或满期日生存。
- 养老年金、教育金、祝寿金等领取期内生存。
- 保证领取期内继续给付且金额可确定。

以下触发条件不得作为普通年金现金流写入:

- 身故。
- 全残、高残、身体全残。
- 重大疾病、恶性肿瘤、中症、轻症、特定疾病。
- 医疗费用发生、住院、门诊、报销。
- 豁免后续保险费。
- 仅描述现金价值、红利、账户价值，但未给出确定领取金额。

## 年金公式归一化

年金写入必须保留真实责任名和计算口径。

常见公式:

- `关爱年金 = 首次交纳的基本责任保险费 × 1%`
- `生存保险金 = 基本责任保险金额 × 9%`
- `生存保险金 = 基本保险金额 × 约定比例`
- `养老年金 = 基本保险金额 × 约定领取比例`
- `满期保险金 = 已交保险费 × 约定比例`
- `满期保险金 = max(基本保险金额, 已交保险费)`
- `祝寿金 = 可选责任保险金额`
- `保证领取保险金 = 保证领取总额 - 已领取年金`

字段要求:

- `coverageType = 现金流`
- `liability` 使用真实责任名。
- `value`, `valueText`, `unit`, `basis`, `formulaText` 尽量结构化。
- `condition` 保留领取时点和频率。
- `sourceRecordId` 和 `sourceExcerpt` 必须存在。
- `responsibilityScope` 必须是 `basic` 或 `optional`。
- optional 责任必须带 `optionalResponsibilityId`，不能默认当作已投保。

## 其他 Lane 的第一期审计规则

### medical_formula

识别:

- `100元`、`1万元` 等免赔额或限额被误当保险金。
- 医疗公式缺少 `实际合理医疗费用 - 已获补偿 - 免赔额`。
- 多个赔付比例缺少社保/非社保、医保内/医保外、计划档位 condition。

第一期只输出报告，不写库。

### critical_illness

识别:

- 疾病关爱金错分现金流。
- 首次/第二次/第三次重疾未拆分。
- 额外给付、多次给付、间隔期、年龄条件缺失。

第一期只输出报告，不写库。

### accident

识别:

- 特定交通意外倍数缺 condition。
- 意外伤残等级比例没有结构化为等级比例。
- 意外医疗被当成普通医疗限额或现金流。

第一期只输出报告，不写库。

### death_life

识别:

- `max(现金价值, 已交保费, 基本保额)` 缺项。
- 年龄分段比例缺 condition。
- 储蓄型身故/全残被误放现金流或反过来漏入寿险责任。

第一期只输出报告，不写库。

### waiver

识别:

- 豁免后续保费被归入现金流。
- 豁免触发条件缺失，例如投保人意外身故/全残、年龄限制。
- 豁免范围缺失，例如主合同、附加合同、基本责任续期保费。

第一期只输出报告，不写库。

## 数据安全

默认数据库:

```text
.runtime/local/policy-ocr.sqlite
```

写库前:

- 确认目标是开发库。
- 创建 SQLite 备份。
- dry-run 输出候选数量、样本和阻塞原因。
- 仅 stage `cashflow_annuity` 高置信候选。

写库后:

- rerun dry-run。
- 目标是本批已写年金候选 `indicatorUpserts: 0`。
- SQL 抽查写入行。
- 若指标影响已有保单，标记相关 `policy_derived_results` stale，触发后续重算流程。

## 输出

终端摘要:

- 总 issue 数。
- 每个 lane issue 数。
- 每个 lane high/medium/low candidate 数。
- 可写候选数。
- 阻塞原因 Top N。

审计报告:

- 按 lane 展示样本。
- 每个样本包含当前指标、建议指标、证据摘录、原因。
- 报告用于人工 review，不作为持久数据源。

## 种子样本

`新华保险 尊享人生年金保险（分红型）` 应在 dry-run 中生成:

- `关爱年金 = 首次交纳的基本责任的保险费 × 1%`
- `生存保险金 = 基本责任保险金额 × 9%`

不得错误写入:

- `投保人意外身故/全残豁免` 不进入现金流。
- `身故或身体全残保险金` 不进入普通年金现金流。
- `祝寿金` 若属于可选责任，保留 optional scope，不默认当作已选现金流。

## 验收标准

### 审计

- 全险种 lane 都能输出 issue 数和样本。
- 审计报告能解释为什么一条记录可写、阻塞或只报告。
- 疾病关爱金不会因为含 `关爱金` 被划入年金现金流。

### 年金写库

- 只写 `cashflow_annuity` 高置信候选。
- 本批写入或替换的候选必须使用真实责任名，不再泛化为 `教育/养老金/两全等返还`。
- 本批写入或替换的候选公式必须保留具体 basis 和比例。
- 写入后 rerun dry-run，本批已写候选收敛。

### 其他险种

- `medical_formula`, `critical_illness`, `accident`, `death_life`, `waiver` 只输出报告。
- 这些 lane 的候选即使 high confidence，也不在第一期写库。

### 测试

Focused tests 覆盖:

- 年金现金流拆分和公式。
- 疾病关爱金不进现金流。
- 医疗免赔额不被当保险金公式。
- 意外伤残等级比例。
- 身故 max 公式。
- 豁免后续保费。
- 可选责任 scope 和 `optionalResponsibilityId`。

## 后续阶段

第二期可以根据第一期审计报告逐个开放写库 lane:

1. `medical_formula`: 先修免赔额误抽和标准医疗费用公式。
2. `critical_illness`: 修疾病关爱金错分、首次/多次给付拆分。
3. `death_life`: 修 max/sum、年龄分段和储蓄型身故。
4. `accident`: 修交通意外倍数和伤残比例。
5. `waiver`: 修豁免责任字段和计算排除规则。

每开放一个 lane，都必须新增对应 focused tests，并保持 dry-run -> backup -> write -> rerun dry-run 的闭环。
