# Agent 语义解析层设计

日期：2026-07-14

状态：已批准，待实施计划

适用范围：钉钉 Agent 当前已支持的产品、家庭、报告、销售和上传意图

## 1. 背景

当前 Hermes 对话运行时能够把用户消息分类为固定意图，并提取少量实体和 `previous_product` 上下文引用。OCR Insurance 随后通过策略路由校验权限、选择受控工具并生成最终回复。

现有契约不足以稳定处理保险业务里的多轮省略、正式名称与简称映射、候选选择和执行条件判断。已观察到两个直接故障：

1. 用户在系统已返回某款产品后询问“主要保啥的呀，这个保险”，解释器识别了产品知识意图，却没有把“这个保险”解析为当前产品，系统随后把整句问题当成产品名称。
2. 用户明确输入“新华人寿保险股份有限公司康健无忧两全保险，这个保险主要保啥的”，解释器把产品缩短成“康健无忧两全保险”，后续匹配没有稳定恢复数据库中的正式产品，最终错误返回资料不足提示。

这两个故障说明意图分类正确并不代表请求已经可以执行。系统还需要独立的上下文指代解析、保险实体标准化、候选消歧和执行就绪判断。

## 2. 目标

第一版语义解析层覆盖当前钉钉 Agent 已支持的以下能力：

- 保险产品查询与比较；
- 家庭列表、家庭概览与当前家庭续问；
- 保障报告和销售报告；
- 销售辅助对话；
- 保单或资料上传入口；
- 现有确认型操作的语义接入，但不改变确认机制。

语义解析层必须做到：

1. 将自然语言转换成受约束、可校验的语义提议；
2. 保留用户原始实体提及，不让模型改写结果覆盖原文；
3. 使用已确认的会话状态解析“这个保险”“这个家庭”“第二款”等指代；
4. 将产品简称、公司名称和家庭名称解析为唯一、已授权的领域实体；
5. 在关键实体缺失、多候选或低置信度时追问，不硬执行；
6. Hermes 不可用时只安全降级处理明确的只读请求；
7. 记录结构化、脱敏、可版本化的解析与决策审计；
8. 保持策略路由、权限、工具白名单、证据边界和写操作确认的最终控制权。

## 3. 非目标

第一版不包含：

- 新增理赔、核保、精算或欺诈调查业务意图；
- 建设通用保险知识图谱；
- 引入向量数据库作为实体真值；
- 允许模型自由生成工具计划；
- 重写现有策略路由、权限系统或领域工具；
- 让 Hermes 或 Direct 解释器直接生成保险责任、金额、赔付或续保结论；
- 将客户、家庭、保单和保险事实写入 Hermes 长期记忆。

向量检索未来可以作为候选召回手段，但不能直接确认正式实体或保险事实。

## 4. 设计原则

### 4.1 模型提出理解，服务端确认实体

Hermes 和 Direct 只描述用户想做什么、提到了哪些原文片段、引用了哪些上下文对象。内部实体 ID、权限、候选唯一性和最终执行决定由 OCR Insurance 确认。

### 4.2 意图置信度不等于执行就绪

系统分别评估意图、实体提及、上下文引用和领域实体解析。即使意图置信度很高，q只要关键实体缺失或存在合理歧义，也必须追问。

### 4.3 平衡型自动解析

- 只读请求在实体唯一、已授权且达到确定性或高置信标准时自动执行；
- 多候选、关键实体缺失或上下文引用不明确时追问；
- 写操作和敏感操作继续进入现有确认流程；
- 降级不能放宽权限、证据或确认要求。

### 4.4 保险事实保持单一权威来源

产品、条款、保单、责任、金额、报告和销售事实继续从 OCR Insurance 当前领域数据读取。语义解析层只解析对象和动作，不补充保险事实。

## 5. 总体架构

语义解析层位于会话加载之后、策略路由之前：

```text
钉钉消息
  -> 加载已验证身份与会话状态
  -> 强信号预解析
  -> Hermes / Direct 语义解释
  -> 语义提议契约校验
  -> 上下文指代解析
  -> 保险领域实体解析
  -> 执行就绪门控
       -> execute      -> 现有策略路由与领域工具
       -> clarify      -> 受控追问
       -> reject       -> 拒绝或权限提示
       -> retry_later  -> 稳定的稍后重试提示
  -> 回复并提交已确认的会话状态
```

现有策略路由继续拥有授权、工具白名单、读写决策和确认机制。语义解析层不能直接调用受保护工具，也不能把模型输出视为授权证据。

## 6. 组件边界

### 6.1 `agent-semantic-contract.mjs`

定义语义契约版本、固定枚举、字段长度、嵌套结构和校验规则。Hermes 与 Direct 必须输出相同契约。

### 6.2 `agent-semantic-preparser.mjs`

识别候选序号、明确编号、上传等高确定性信号。预解析器只产生结构化提示，不调用工具、不决定保险事实。

### 6.3 `agent-semantic-resolver.service.mjs`

编排上下文解析、领域实体解析和执行就绪判断。它接收已验证身份、原始消息、语义提议与会话状态，输出解析后的语义帧。

### 6.4 `agent-product-entity-resolver.service.mjs`

使用现有产品目录、正式名称、备案名称、条款编号、保险公司别名和已审核产品别名解析产品。产品知识检索只消费已解析的产品，不再同时承担开放式实体识别。

### 6.5 `agent-family-entity-resolver.service.mjs`

只在当前用户已授权的家庭集合内解析家庭名称、当前家庭和家庭候选。模型或渠道传入的内部 ID 不能作为授权证据。

### 6.6 `agent-semantic-readiness.service.mjs`

根据意图所需字段、实体唯一性、授权结果、操作风险和运行时降级状态，输出 `execute`、`clarify`、`reject` 或 `retry_later`。

### 6.7 `agent-conversation-runtime.service.mjs`

该服务保持为编排层：加载会话、调用解释器、调用语义解析器、交给策略路由、保存已确认状态。实体匹配和门控规则不继续堆入该文件。

## 7. 语义提议契约

Hermes 或 Direct 输出 `SemanticProposal`。示例：

```json
{
  "semanticContractVersion": 1,
  "intent": "insurance_product_knowledge",
  "operation": "read",
  "queryAspects": ["main_responsibilities"],
  "mentions": [
    {
      "type": "insurer",
      "rawText": "新华人寿保险股份有限公司"
    },
    {
      "type": "product",
      "rawText": "康健无忧两全保险"
    }
  ],
  "references": [
    {
      "type": "current_product",
      "rawText": "这个保险"
    }
  ],
  "requestedSteps": ["lookup"],
  "confidence": {
    "intent": 0.98,
    "mentions": 0.94,
    "references": 0.92
  }
}
```

契约约束：

- `intent` 只能从当前发布的意图列表选择；
- `operation` 只能从允许的读写操作中选择；
- `queryAspects`、`mention.type`、`reference.type` 和 `requestedSteps` 均为固定枚举；
- `mentions[].rawText` 必须是用户原文中的连续片段；
- `references` 只描述指代关系，不携带内部实体 ID；
- `requestedSteps` 只表达用户请求，不是已授权工具计划；
- 模型不能输出保险事实、权限判断、内部用户 ID、家庭 ID、产品 ID 或最终工具名称；
- 未知字段、超长字段、非法枚举和无法在原文定位的提及使整个提议无效，不做宽松修补。

第一版沿用当前固定大意图集合，并用 `queryAspects` 描述细分需求，避免为等待期、免责、续保等维度创建大量独立意图。

## 8. 解析后的语义帧

服务端生成 `ResolvedSemanticFrame`：

```json
{
  "semanticContractVersion": 1,
  "intent": "insurance_product_knowledge",
  "operation": "read",
  "queryAspects": ["main_responsibilities"],
  "resolvedEntities": {
    "insurer": {
      "canonicalName": "新华保险",
      "matchType": "approved_alias",
      "confidence": 1
    },
    "product": {
      "canonicalProductId": "product_xxx",
      "officialName": "新华人寿保险股份有限公司康健无忧两全保险",
      "matchType": "company_scoped_alias",
      "confidence": 0.99
    }
  },
  "ambiguities": [],
  "missingFields": [],
  "decision": "execute",
  "decisionReason": "unique_authorized_product"
}
```

该对象明确区分模型提议和服务端确认结果。策略路由只消费解析后的意图、操作与实体，不直接消费未经确认的模型实体。

## 9. 会话状态

会话状态从单一“上一产品”扩展为类型化任务状态：

```json
{
  "activeIntent": "insurance_product_knowledge",
  "activeEntities": {
    "product": {
      "canonicalProductId": "product_xxx",
      "displayName": "新华人寿保险股份有限公司康健无忧两全保险",
      "updatedAt": 1783955157496
    },
    "family": null
  },
  "candidateSets": {
    "product": [],
    "family": []
  },
  "pendingClarification": null,
  "lastCompletedAction": {
    "intent": "insurance_product_knowledge",
    "entityType": "product"
  }
}
```

状态规则：

- 只保存服务端已确认实体，不保存模型未确认猜测；
- 显式新实体覆盖当前同类型实体；
- 产品和家庭状态分别维护，不能互相覆盖；
- 上下文引用只能使用仍在 TTL 内的对应实体；
- 候选序号只在对应 `pendingClarification` 的有效期内生效；
- Hermes Session 保存语言连续性，但不是业务实体事实来源；
- 保险事实不进入会话状态，每次从当前领域数据读取。

持久化优先复用现有 `agent_conversations` 和 `agent_conversation_entities`。`agent_conversation_entities` 保存已确认的当前实体和候选；会话负载保存任务状态、待追问状态和 Hermes Session ID。旧 `product` 与 `productCandidates` 在迁移期兼容读取，新的写入统一使用类型化状态。

## 10. 解析优先级

每轮消息按以下证据强度处理：

1. 有效待追问状态中的候选序号选择；
2. 本轮明确提及的公司、产品或家庭；
3. 本轮明确的上下文指代；
4. 当前任务中仍有效的已确认实体；
5. 无法确定时生成追问。

如果本轮同时出现明确产品名和“这个保险”，明确产品名优先，“这个保险”作为同一对象的辅助信号。原始用户问题始终保留；解析层把确认后的实体单独传给路由和工具，不用重写后的问题覆盖原文。

## 11. 产品实体解析

产品解析器按以下顺序工作：

1. 正式名称完全匹配；
2. 条款编号或备案名称完全匹配；
3. 保险公司范围内的已审核别名匹配；
4. 保险公司范围内的规范化名称匹配；
5. 字符检索召回候选；
6. 可选语义召回补充候选。

匹配结果类型固定为：

- `exact_official_name`；
- `filing_name`；
- `approved_alias`；
- `company_scoped_normalized`；
- `unique_high_confidence`；
- `ambiguous`；
- `not_found`。

决策规则：

- 前四类唯一匹配可直接执行只读请求；
- `unique_high_confidence` 只适用于只读请求，第一版要求归一化解析置信度不低于 0.90，且第一、第二候选置信度差不低于 0.15；阈值调整必须随解析规则版本发布并进入审计；
- 多个合理候选必须让用户选择，不能静默选择；
- 没有候选时询问保险公司、正式名称、版本、计划或正式资料；
- 不允许把整句问题退化为产品名称；
- 找到正式产品但没有可核验证据时，可以确认产品身份，但不能生成保险事实，只能返回证据不足提示。

产品别名必须来自审核或确定性生成规则。模型临时产生的简称不能直接写入正式别名表。

## 12. 家庭、报告与销售解析

### 12.1 家庭

家庭实体只在当前用户已授权集合内解析：

- 精确名称或已确认别名唯一匹配时执行；
- 多个同名或近似家庭时展示脱敏候选；
- 当前家庭引用不存在或已过期时追问；
- 外部输入的内部家庭 ID 不能作为授权证据；
- 未授权家庭不得进入候选、会话状态或工具输入。

### 12.2 报告

报告语义由“报告类型 + 目标家庭 + 用户动作”组成。语义层只解析对象和动作；实际报告是否可查看、是否需要刷新以及是否允许生成，由现有策略路由和报告服务决定。

### 12.3 销售辅助

销售辅助可以继承当前已授权家庭，但不能从 Hermes Memory 读取或写入未经治理的客户事实。家庭销售记忆继续遵循现有确认、双时间和权限规则。

### 12.4 上传

上传相关高确定性表达可由预解析器识别，但渠道附件策略、安全链接、企业开关和原件隐私边界保持不变。

## 13. 上下文指代

第一版支持以下引用类型：

- `current_product`：这个保险、它、该产品、刚才那款；
- `current_family`：这个家庭、他家、刚才那个客户家庭；
- `candidate_index`：第一款、第二个、选 2；
- `previous_result`：继续、再详细一点、导出一下；
- `comparison_left` 与 `comparison_right`：产品比较两侧对象。

“他”等跨类型代词不能默认解释为产品或客户。解析器结合当前意图和活动实体判断；如果产品和家庭两种解释都合理，则生成追问。

## 14. 执行就绪门控

语义层统一输出：

- `execute`：意图明确、关键实体唯一、授权有效且当前操作可以交给策略路由；
- `clarify`：缺少关键参数、存在多候选或引用不明确；
- `reject`：越权、非法或不支持的请求；
- `retry_later`：依赖服务不可用且无法安全降级。

具体规则：

- 高意图置信度但关键实体缺失时 `clarify`；
- 实体唯一但意图不明确时 `clarify`；
- 只读、实体唯一且已授权时 `execute`；
- 写操作语义清楚时交给现有确认流程，不能直接写入；
- 越权或不支持时 `reject`；
- Hermes 故障但规则或 Direct 可以唯一确定只读请求时允许安全执行；
- Hermes 故障且仍有歧义时 `clarify` 或 `retry_later`。

## 15. 降级与错误处理

运行时降级顺序：

```text
Hermes 成功且契约有效
  -> 使用 Hermes 提议

Hermes 超时、失败或输出非法
  -> 记录失败原因
  -> Direct 使用相同契约解析

Direct 也失败
  -> 使用强信号规则

规则能唯一确定明确只读请求
  -> 继续执行

仍有歧义、涉及写入或敏感操作
  -> clarify 或 retry_later
```

错误边界：

- Hermes 输出非法字段时拒绝整个输出，不做猜测式修补；
- 产品目录不可用时不使用模型记忆猜产品，返回稍后重试；
- 会话版本冲突时重新加载并重试一次，仍冲突则不覆盖更新后的状态；
- 候选选择过期时提示用户重新选择；
- 语义审计写入失败时，只读请求可以发出不含业务内容的运行错误日志后返回；写操作继续遵循现有强审计和确认要求；
- 降级路径不得放宽权限、来源核验、隐私或确认规则。

## 16. 结构化脱敏审计

每轮记录独立的语义审计事件，示例：

```json
{
  "semanticContractVersion": 1,
  "runtime": "hermes",
  "fallbackReason": null,
  "intent": "insurance_product_knowledge",
  "operation": "read",
  "redactedMentions": [
    {
      "type": "product",
      "value": "康健无忧两全保险"
    }
  ],
  "resolution": {
    "productMatchType": "company_scoped_alias",
    "candidateCount": 1,
    "selectedCanonicalProductId": "product_xxx"
  },
  "decision": "execute",
  "decisionReason": "unique_authorized_entity"
}
```

审计要求：

- 保存实际使用的 `hermes`、`direct` 或 `rule`；
- 保存降级原因、候选数量、匹配类型、最终决策和规则版本；
- 不重复保存完整聊天内容；
- 手机号、保单号、客户姓名等按现有隐私规则掩码或令牌化；
- 不保存 Hermes 完整 Prompt、完整上下文、模型思维过程或保险原件；
- 语义契约与解析规则带版本号，以支持回放和上线前后比较。

语义审计发生在策略路由之前，不能只依赖现有路由审计。持久化应通过窄 SQLite store 方法完成，不使用全状态持久化。

## 17. 测试策略

### 17.1 契约测试

- Hermes 与 Direct 输出通过同一校验器；
- 非法意图、额外字段、内部 ID 和超长文本被拒绝；
- `rawText` 必须可在原始用户消息中定位；
- 模型输出的保险事实或工具名称不能进入执行。

### 17.2 实体解析测试

- 公司全称、简称和审核别名映射到同一公司；
- 产品正式名称、简称、备案名称和条款编号正确归一；
- 同名跨公司产品先按公司范围消歧；
- 多候选不静默选择；
- 未授权家庭不进入候选结果。

### 17.3 上下文测试

- “这个保险”引用当前产品；
- “这个家庭”引用当前家庭；
- “1”和“第二款”只在有效候选集合中生效；
- 新的明确实体覆盖旧实体；
- 过期状态要求重新确认；
- 产品状态和家庭状态互不覆盖。

### 17.4 门控与降级测试

- 高意图置信度但缺关键实体时追问；
- 唯一高置信只读实体可以执行；
- 写操作仍进入确认流程；
- Hermes 故障后只允许安全降级；
- 产品目录故障时不使用模型知识回答保险事实。

### 17.5 钉钉端到端回归

必须覆盖：

```text
用户：新华人寿保险股份有限公司康健无忧两全保险，这个保险主要保啥的
预期：解析到正式产品并返回已有官方证据
```

```text
系统：已返回康健无忧两全保险
用户：主要保啥的呀，这个保险
预期：继承当前产品，不把整句话当产品名
```

还应覆盖产品比较后的“第二款等待期呢”、家庭查询后的“这个家庭保障缺口呢”、多个同名家庭、Hermes 故障后的明确只读查询，以及 Hermes 故障后的模糊或写操作。

### 17.6 语义回归语料

维护脱敏、可版本化的评估夹具，覆盖正式表达、口语、省略、错别字、上下文指代、公司与产品同名歧义、多意图、参数补充、越权请求和 Hermes/Direct 差异。评估语料不进入生产长期记忆。

## 18. 分阶段上线

### 阶段一：影子解析

新语义层运行并写脱敏审计，但不改变现有路由结果。比较旧候选与新解析结果，确认错误类型和候选分布。

### 阶段二：产品与家庭只读流量

接管产品知识、产品比较、家庭列表和家庭概览。任何未达到执行门槛的请求都转为追问。

### 阶段三：报告与销售流量

接管保障报告、销售报告和销售辅助语义。报告权限、刷新规则和家庭销售记忆治理保持不变。

### 阶段四：全部当前意图

上传和现有确认型操作接入统一语义层，但不改变安全链接、隐私边界和确认机制。

每个阶段都通过运行时配置支持回退到旧解析链路。回退不迁移、不删除业务事实。

## 19. 成功标准

上线前必须满足：

- 两个已发现的康健无忧故障场景通过；
- 全部确定性实体、上下文、授权和门控测试通过；
- 未授权家庭不会进入候选、会话状态或工具调用；
- 多候选不会静默执行；
- Hermes 故障不会放宽写操作或敏感操作；
- 每次请求可以审计实际使用的 `hermes`、`direct` 或 `rule`；
- 语义审计不重复保存完整聊天或未脱敏敏感字段；
- 产品答案继续只来自受控证据，不从解释器记忆补充事实；
- 影子阶段发现的解析差异已经分类，并有明确的执行、追问或回退结果。

## 20. 与现有设计的关系

本设计细化并扩展《Hermes 对话运行时接入 OCR Insurance 设计》中的语言层与 OCR 业务状态边界：

- Hermes 仍是主对话解释运行时；
- OCR Insurance 仍是身份、权限、业务事实、工具执行和审计的唯一权威系统；
- 新增的语义解析层位于 Hermes 提议和现有策略路由之间；
- Direct 保持为安全降级解释器，不成为第二个业务权威；
- 本设计不改变钉钉原件上传、家庭销售记忆或保险证据治理的既有安全边界。
