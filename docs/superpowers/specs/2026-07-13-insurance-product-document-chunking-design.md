# 保险产品资料切片设计

日期：2026-07-13  
状态：设计稿  
适用范围：正式保险条款、产品说明书、费率/保障计划表、投保规则、健康告知、理赔资料、服务手册、培训课件和营销资料

## 1. 目标

保险产品知识切片不能只追求语义相似度。它必须同时保证：

1. 产品、版本和资料来源不混淆；
2. 责任、条件、金额、比例、免赔额和例外不被拆散；
3. 表格中的表头、计划列、数据行和脚注能一起被解释；
4. 回答中的每个合同事实都能回到正式资料、页码和原文；
5. 切片不足时可回到同一份官方全文，而不是把“未命中”误判为“产品不存在”。

本设计不以固定字符数作为主边界。固定 Token 上限只用于逻辑单元过长后的二次拆分。

## 2. 资料分级

切片前先确定 `document_type` 和 `source_authority`。不同资料不能使用同一套解释规则。

| 资料类型 | 主要用途 | 可支持的结论 | 默认权威等级 |
| --- | --- | --- | --- |
| 正式利益条款/基本条款 | 合同责任、免责、等待期、免赔额、赔付比例、续保 | 确定性合同事实 | `insurer_official` |
| 保障计划表/费率表 | 不同计划的金额、比例、限额、价格 | 确定性计划事实 | `insurer_official` |
| 产品说明书/投保须知 | 投保规则、计划说明、销售状态线索 | 经条款复核的产品事实 | `insurer_official` |
| 健康告知/核保规则 | 告知问题、投保限制、核保材料 | 投保规则，不承诺核保结果 | `insurer_official` 或 `approved_company_material` |
| 理赔申请/服务手册 | 理赔材料、流程、服务范围 | 流程事实，不扩大保险责任 | 同来源等级 |
| 培训课件/销售手册 | 产品定位、卖点、适用场景、话术 | 营销解释，不能覆盖条款 | `approved_company_material` |
| 新闻、测评、第三方网页 | 产品发现和线索 | 不支持确定性合同结论 | `open_web_reference` |

同一问题出现冲突时，正式条款和批单优先于说明书，说明书优先于培训资料，第三方资料只作为发现线索。

## 3. 总体模型：四层知识单元

### 3.1 文档层 `document`

保留整份原文件及身份信息，不直接参与普通回答，但用于全文兜底和重新切片。

必备字段：

- `canonical_product_id`
- `product_version_id`
- `company`
- `official_product_name`
- `document_type`
- `source_authority`
- `source_url`
- `effective_from` / `effective_to`
- `filing_code` / `clause_code`
- `content_hash`
- `parser_version` / `chunking_version`

### 3.2 章节父块 `section_parent`

父块代表一个完整合同章节或业务章节，例如“第六条 保险责任”“4. 医疗费用保险金计算方法”。它用于补充上下文，不能替换真正命中的子块。

建议大小：800～2000 Token。超长章节按款、项拆分为多个父块，并保留共同的 `clause_path`。

### 3.3 可检索证据块 `evidence_chunk`

这是 BM25、向量和重排的主要对象。按业务语义分为：

- `clause`：责任、免责、续保、投保范围等条款；
- `fact`：等待期、免赔额、限额、比例、年龄等可直接回答的事实；
- `formula`：赔付公式及公式变量定义；
- `definition`：疾病、医院、既往症等释义；
- `table`：带表头和脚注的表格行组；
- `list_item`：药品、疾病、医院或材料清单中的单项；
- `process`：投保、保全、理赔和服务流程；
- `claim`：经过审核的产品卖点或适用场景。

建议大小：正式条款 300～800 Token，培训资料 200～500 Token。只有单个逻辑单元超过上限时才按完整句、完整款项拆分。

### 3.4 结构化事实 `product_fact`

金额、比例、期限和枚举条件不能只依赖文本召回。切片时同步抽取结构化事实，并回连证据块：

```json
{
  "field_key": "annual_deductible",
  "value": 10000,
  "unit": "CNY",
  "scope": {
    "plan": "计划一",
    "responsibility": "基本责任",
    "policy_year": "每个保险期间"
  },
  "exceptions": ["可选责任不适用年度免赔额"],
  "evidence_chunk_id": "...",
  "status": "confirmed"
}
```

结构化事实负责精确回答，证据块负责解释条件和引用原文。

## 4. 正式条款切片规则

### 4.1 边界优先级

按以下顺序识别边界：

1. 条：`第六条 保险责任`；
2. 款：`4. 医疗费用保险金计算方法`；
3. 项：`（1）年度免赔额`；
4. 子项：`①住院医疗费用保险金`、`A.门诊肾透析`；
5. 完整句；
6. Token 硬上限。

禁止在以下位置切断：

- 标题与紧随其后的正文之间；
- 计算公式与变量定义之间；
- 责任名称与触发条件、给付方式之间；
- “但、除外、另有约定、前述、不适用”等限制词前后；
- 表格表头与数据行之间；
- 跨页未结束的句子或款项之间。

### 4.2 责任块的最小完整性

每个责任块尽量包含：

`责任名称 + 触发条件 + 保障范围 + 给付方式 + 限额/比例 + 关键限制`

如果限制位于相邻款，主块保留正文，同时通过 `required_neighbor_chunk_ids` 绑定限制块。回答时必须一起装配。

### 4.3 数值事实完整性门

等待期、免赔额、赔付比例、给付限额、投保年龄、保证续保期等关键字段，发布前检查：

- 是否有明确数值；
- 是否有单位；
- 是否有计划/责任/年度等适用范围；
- 是否存在例外；
- 是否能回到页码和条款路径。

缺少任何关键部分时，事实状态为 `incomplete`，不能生成确定性回答；检索时自动扩展相邻块或读取同一官方全文。

## 5. 表格切片规则

表格不能只保存展平后的页面文字。

### 5.1 普通保障计划表

每个表格块必须重复携带：

- 表名；
- 多级表头；
- 计划列名称；
- 当前行组；
- 表下注释；
- 所属产品版本。

例如“年度免赔额 | 计划一 1万元 | 计划二 2万元”应形成一个独立事实块，同时抽取两条 `product_fact`。

### 5.2 跨页表格

后续页没有新表头但列结构一致时，继承前页表头，并记录：

- `header_inherited_from_page`
- `page_start` / `page_end`
- `table_id`
- `row_start` / `row_end`

### 5.3 清单类表格

疾病、药品、医院、材料和职业清单按“单项或小行组”建立 `list_item`，同时保留清单名称、版本、适应症和脚注。不能把几百项清单整体塞入一个块。

## 6. 其他资料的切片规则

### 产品说明书和投保须知

按“产品定位、投保年龄、保险期间、缴费方式、等待期、续保、保障计划、责任概览、除外提示”切分。合同数字仍需关联正式条款证据。

### 健康告知和核保资料

按问题编号切分，每块保留适用人群、时间范围、疾病/检查项和补充说明。结论只能描述告知要求，不得生成承保承诺。

### 理赔和服务资料

按场景切分：报案、材料、审核、给付、时效、服务入口。理赔流程块必须与合同责任块分开，避免把“可申请”误写成“必赔”。

### 培训课件和营销资料

保持页内切片，按“产品优势、适用人群、异议处理、案例、话术”分类。每块标记 `non_contractual=true`；出现合同数字时必须关联官方证据，否则只作为待核验主张。

## 7. 上下文和元数据

每个证据块至少保存：

```text
产品身份：company / canonical_product_id / product_version_id
来源身份：document_id / document_type / source_authority / source_url
位置：page_start / page_end / clause_path / table_id / row_range
业务身份：chunk_type / responsibility / fact_keys / plan_names / topics
关系：parent_chunk_id / previous_chunk_id / next_chunk_id / required_neighbor_chunk_ids
治理：review_status / effective_from / effective_to / content_hash / parser_version / chunking_version
质量：boundary_confidence / numeric_completeness / ocr_quality / review_required
```

`contextual_prefix` 可包含公司、产品、版本、资料类型、章节和页码，但不能用前缀替代结构化过滤。

## 8. 查询时的证据装配

不同问题使用不同检索计划：

| 问题类型 | 首选 | 补充证据 |
| --- | --- | --- |
| 免赔额、等待期、比例、年龄 | confirmed `product_fact` | 对应 fact/clause/table + 例外块 |
| 保险责任 | 责任卡 + clause | 触发条件、限额、免责/限制邻块 |
| 条款解释 | 精确条款块 | 父章节、释义和引用条款 |
| 产品对比 | 两款同字段 facts | 两边同版本官方证据、缺失项 |
| 产品优势 | confirmed facts + approved claims | 正式条款限制和培训资料 |
| 理赔问题 | 责任块 + 理赔流程 | 免责、材料和事实输入要求 |

装配顺序：

1. 锁定公司、产品和版本；
2. 查询结构化事实；
3. 召回命中子块；
4. 加入必需限制块和表格脚注；
5. 必要时扩展父章节，但不替换命中子块；
6. 检查数值、适用范围和来源是否完整；
7. 不完整时读取同一官方全文并临时切片；
8. 仍不完整才向用户追问。

## 9. 与现有实现的差距

现有 `product-chunker.service.mjs` 已具备条/款标题、跨页延续、父子块、表格块、前后块和业务主题，是可继续演进的基础。主要差距是：

1. 父块目前按页面生成，应逐步增加章节父块；
2. 关键数值没有形成可发布的结构化事实；
3. 公式、事实、释义、限制和清单尚未区分 chunk type；
4. 相邻块只有前后关系，没有“回答必须携带”的限制关系；
5. 表格脚注、跨页表头和行级事实的绑定不足；
6. 切片发布前没有数值完整性门；
7. 正式条款和培训资料虽然有来源等级，但检索策略仍需进一步隔离。

## 10. 分阶段实施

### 第一阶段：解决精确事实漏答

- 增加 `fact`、`formula`、`definition`、`clause`、`table` 类型；
- 抽取等待期、免赔额、比例、限额、年龄和续保事实；
- 增加数值完整性检查；
- 保留当前全文兜底；
- 用康健长佑等真实产品建立回归集。

### 第二阶段：提高条款解释正确性

- 章节父块替代页面父块；
- 建立责任—限制—释义—表格脚注关系；
- 支持跨页表格表头继承；
- 查询装配时强制加入必要限制块。

### 第三阶段：产品比较和多资料融合

- 建立产品版本隔离；
- 同字段、同计划维度比较；
- 正式条款与已审核培训资料分通道召回；
- 增加错版本污染和营销资料越权检测。

## 11. 验收指标

至少建立 100 条黄金问题，覆盖医疗险、重疾险、寿险、年金险和意外险。发布门建议为：

- 精确字段 Recall@5 ≥ 98%；
- 产品和版本准确率 ≥ 99%；
- 数值与单位一致率 = 100%；
- 必要限制遗漏率 ≤ 1%；
- 官方引用页码准确率 ≥ 98%；
- 培训资料冒充合同事实次数 = 0；
- 无证据确定性断言次数 = 0；
- 全文兜底成功率和 P95 延迟持续监控。

首批回归问题至少包括：

- 康健长佑计划一、计划二的年度免赔额；
- 可选责任是否适用免赔额；
- 未经医保结算时的赔付比例；
- 一般医疗与特定疾病医疗的年度限额；
- 表格数据与正文不一致时采用哪一项；
- 同名不同版本产品不得互相引用。
