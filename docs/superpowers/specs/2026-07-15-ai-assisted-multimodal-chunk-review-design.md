# AI 辅助多模态切片审核与修正闭环设计

日期：2026-07-15

状态：待用户复核

适用范围：产品条款、说明书、费率表、保障计划、培训课件、图片、扫描件、PPT 和其他进入产品知识 RAG 的资料

## 1. 背景与目标

现有产品知识流程已经支持上传、解析、切片、候选索引、查看 RAG 内容、发布、拒绝、回滚和下架，但审核仍以整份资料的“通过/拒绝”为主。审核员可以看到切片，却不能稳定完成以下工作：

- 将切片与原始页面或图片区域逐一比对；
- 发现原图有内容但切片遗漏；
- 删除误入切片的页眉、页脚、二维码或其他产品内容；
- 修正被错误拆开的条件、数值、限制、脚注和跨页内容；
- 将自然语言审核意见转换为可预览、可执行的修正计划；
- 保存修正原因、操作过程和新旧版本，供再次审核与回滚。

本设计建立以下闭环：

```text
上传原始资料
→ OCR、版面识别和确定性切片
→ 规则质检
→ 多模态大模型预审
→ 人工复核问题队列
→ 审核意见解析为结构化修正计划
→ 人工确认或微调
→ 生成新候选索引版本
→ 自动检索验证
→ 人工最终发布
```

成功标准是：审核员不修改代码，也能处理单份资料的 OCR、文字、图片、表格、边界、遗漏、多余内容、跨页关系和产品绑定问题；所有修正均进入 SQLite，并能追溯到原文件、原图区域、修正人和索引版本。

## 2. 设计原则

### 2.1 原始证据不可变

原始文件、原始页面、原始 OCR 和当前正式索引不能被人工修正直接覆盖。修正只作用于新的候选版本。发布时原子切换候选版本；旧正式版本保留为 `superseded`，支持回滚。

### 2.2 模型预审，人工决定

大模型可以发现问题、解释原因、提出操作和生成验证问题，但不能：

- 修改原始文件；
- 直接改写正式保险条款；
- 直接发布、下架或替换正式索引；
- 在没有来源页和来源区域时补造合同事实；
- 把模型生成的背景说明冒充原文证据。

### 2.3 保险语义完整性优先于固定长度

切片不以固定 Token 数作为主要边界。每个保险证据单元优先保持以下结构完整：

```text
责任或事项名称
+ 适用对象
+ 触发条件
+ 给付或处理方式
+ 金额、比例或期限
+ 限制与例外
+ 适用计划、产品和版本
```

Overlap 只作为句子边界的最后兜底，不用于替代条款、表格、图片和业务关系识别。

### 2.4 原文、检索背景和结构化事实分层

- `content`：可引用的来源原文，不由模型自由改写；
- `contextualPrefix`：公司、产品、版本、资料类型、章节、页码等确定性背景；
- `generatedContext`：模型补充的检索背景，单独保存模型版本和审核状态；
- `product_facts`：等待期、免赔额、比例、限额等结构化候选事实，必须回连原文证据。

### 2.5 单份资料修正与通用算法升级分开

单份资料失败优先通过框选、合并、拆分、排除、补充关系和重新 OCR 解决。同类失败在多个文档反复出现，才进入通用解析或切片代码优化。

## 3. 审核对象：多模态来源元素

每个页面的 `layout_json` 增加稳定的来源元素列表。第一版不新建另一套页面事实表，继续以 `product_document_pages` 为页面事实来源：

```json
{
  "elements": [
    {
      "id": "el_p8_014",
      "kind": "text",
      "bbox": [90, 620, 1050, 710],
      "text": "未经基本医疗保险结算的，给付比例为60%",
      "ocrConfidence": 0.96
    },
    {
      "id": "el_p8_015",
      "kind": "business_image",
      "bbox": [80, 220, 1080, 600],
      "assetRef": "page_8_crop_015",
      "caption": "保障计划对比表"
    }
  ]
}
```

来源元素类型至少包括：

- `text`：正文、标题、脚注；
- `table`：表格及行列结构；
- `business_image`：保障图、流程图、费率截图等业务图片；
- `decoration`：Logo、背景图、装饰图；
- `header_footer`：页眉、页脚、页码；
- `qr_contact`：二维码、电话、公众号等非知识内容。

每个候选切片在 `payload.sourceRegions` 保存来源锚点：

```json
{
  "sourceRegions": [
    {
      "pageNo": 8,
      "bbox": [90, 620, 1050, 710],
      "elementIds": ["el_p8_014"]
    }
  ]
}
```

因此系统可以计算：哪些来源元素已被切片覆盖、哪些有效元素遗漏、哪些切片内容找不到来源，以及一个切片是否错误跨越多个产品或章节。

## 4. 图片和表格如何切片

### 4.1 业务图片独立成块

只有包含业务信息的图片进入知识切片。Logo、装饰图、二维码和纯宣传背景默认排除，但保留排除结果供人工检查。

业务图片块同时保存：

```text
原图裁剪
+ 页面和坐标
+ OCR文字
+ 图片说明
+ 表格行列或流程节点
+ 所属公司、产品、版本和章节
+ 相关文字块
+ 结构化事实候选
```

图片描述用于召回，原图裁剪用于人工审核和回答证据追溯。图片描述不能替代正式条款原文。

### 4.2 表格按可解释行组切片

表格块必须携带表名、多级表头、计划列、当前行组和脚注。表格图片、OCR 展平文本和结构化单元格三者保持关联。脚注不得因为视觉位置在表格下方而丢失。

### 4.3 图片漏切

审核员可以在原图上框选遗漏图片。系统重新执行图片分类、OCR 或图像理解，生成候选图片块。所有识别结果先预览，人工确认后才进入新候选版本。

## 5. AI 预审：四层审核

所有候选资料都经过模型预审。确定性规则先生成页面元素、覆盖图和质量告警，模型在这些证据上做语义判断。长文档按页面窗口分批审核，最后再做整份文档汇总，避免单次上下文过大。

### 5.1 来源覆盖审核

对比原图/OCR元素与候选切片：

- 有效正文、脚注、表格和业务图片是否都有切片覆盖；
- 是否有切片内容找不到来源区域；
- 是否有页眉、页脚、二维码和联系方式进入知识块；
- 是否有整页或某个产品页面范围未被处理。

### 5.2 语义完整性审核

检查责任、条件、数值、单位、适用范围和限制是否被拆散。例如“给付比例60%”如果没有携带“未经基本医疗保险结算”的条件，应判定为高风险，而不是因为文字存在就通过。

### 5.3 关系完整性审核

检查命中块是否建立必要关系：

- `previous` / `next`：自然相邻；
- `parent`：章节上下文；
- `required_context`：回答时必须携带的限制、例外、脚注或变量定义；
- `defines`：释义关系；
- `applies_to`：适用责任、计划、产品或版本；
- `conflicts_with`：与另一来源存在冲突，必须人工判断。

句子窗口只作辅助。保险条款优先使用语义关系扩展，因为免责、释义和脚注可能不在相邻页面。

### 5.4 检索验证

系统根据切片和事实候选生成可验证问题，例如：

```text
未经医保结算时赔付比例是多少？
计划三是否适用于所有地区？
年度免赔额是否适用于可选责任？
```

验证不以模型答案是否流畅为准，而检查 Top-K 证据是否同时包含：主事实、适用条件、限制块、正确产品版本和来源页。只命中数值但漏掉条件时，候选版本仍不合格。

## 6. AI 预审输出契约

模型必须返回结构化问题，不接受只有自然语言总结的结果：

```json
{
  "decision": "human_review_required",
  "issues": [
    {
      "type": "semantic_incomplete",
      "severity": "high",
      "confidence": 0.94,
      "pageNos": [8],
      "sourceRegions": [
        { "pageNo": 8, "elementIds": ["el_p8_014"] }
      ],
      "affectedChunkIds": ["chunk_008_03"],
      "reason": "60%给付比例缺少未经医保结算的适用条件",
      "missingElements": ["condition"],
      "proposedOperations": [
        { "type": "add_source_elements", "elementIds": ["el_p8_014"] },
        { "type": "create_required_context" }
      ]
    }
  ]
}
```

服务端必须验证：引用的页码、元素 ID、切片 ID 和产品 ID 真实存在。没有有效来源锚点的合同事实建议不能进入修正计划。

上传资料中的文字视为不可信数据。模型提示明确禁止执行资料中的指令，输出必须通过固定 JSON Schema 校验，防止文档提示注入改变审核或发布动作。

## 7. 人工审核工作台

工作台采用三栏布局：

```text
左：AI问题队列和页码
中：原图、OCR元素和来源覆盖高亮
右：当前切片、AI判断、不通过说明、修正计划和新旧对比
```

左侧按严重性、问题类型和置信度排序，但“模型认为正常”的切片仍可查看。中间支持缩放、翻页、查看原图/OCR覆盖层和跨页来源。右侧支持：

- 编辑 OCR 文字；
- 从光标处分割；
- 与上一块或下一块合并；
- 从已有来源元素补入遗漏内容；
- 在原图上框选遗漏区域并重新 OCR/识图；
- 排除当前切片中的多余区域；
- 扫描并预览整份资料的重复页眉、页脚或二维码；
- 建立或删除 `required_context` 等关系；
- 修正产品、版本、章节、责任或计划绑定；
- 标记“模型判断错误”。

“编辑 OCR 文字”只生成修正覆盖层和新候选内容，不更新原始 `raw_text`。审核界面始终可以在原始 OCR、修正文本和原图之间切换比较。

“不通过”拆分为：

- `退回修正`：资料有价值，进入修正闭环；
- `整份作废`：重复、无效、错误资料，不再处理。

## 8. 审核意见到修正计划

审核员采用“结构化原因 + 自然语言说明”。结构化原因至少包括：

```text
OCR错误
内容缺失
内容多余
切片边界错误
语义不完整
图片漏切或错切
表格结构错误
跨页关系缺失
产品或版本绑定错误
资料无效
```

模型把说明解析成有限操作集合：

```text
edit_ocr_text
split_chunk
merge_chunks
add_source_elements
remove_source_elements
reclassify_image
exclude_region_current_chunk
exclude_repeated_regions_document
rebind_product
rebind_version
create_relation
remove_relation
regenerate_context
regenerate_fact_candidates
```

模型不能生成任意数据库操作。修正计划必须先展示：目标对象、来源区域、修改前后、影响切片数、影响事实数和预计影响页。人工确认后，服务端在事务中生成新候选版本。

## 9. 排除内容的作用范围

审核员排除多余内容时可选择：

### 9.1 仅当前切片

用于一次性串入的其他段落、框选过大的图片或偶发无关内容。其他页面不受影响。

### 9.2 当前资料的重复区域

用于多页重复的页眉、页脚、Logo、二维码或联系方式。系统先展示所有匹配页面和区域，人工确认后才批量排除。规则只作用于当前文档，不自动扩散到其他资料。

### 9.3 产品页面范围纠正

另一产品内容串入时，不使用简单文本排除，而是重新设置页面范围和产品绑定，重建受影响切片。

## 10. 持久化与审计

继续复用：

- `product_documents`：文档和总体状态；
- `product_document_blobs`：原始文件；
- `product_document_pages`：页面、OCR、版面和来源元素；
- `knowledge_chunks`：候选、正式和历史切片版本；
- `product_facts` / `product_fact_evidence`：结构化事实及证据；
- `product_document_links`：产品和版本范围。

新增三张窄表：

### 10.1 `product_document_review_runs`

保存规则质检、模型预审和检索验证的运行信息：文档、候选索引版本、模型/规则版本、状态、开始结束时间、统计摘要和错误原因。

### 10.2 `product_document_review_issues`

保存每个问题的类型、严重性、置信度、来源区域、受影响切片、原因、模型建议、人工结论和解决状态。

### 10.3 `product_document_corrections`

保存审核意见和最终操作：来源问题、候选版本、操作类型、修改前后、影响范围、操作者、确认时间和结果版本。

这些表是审核和修正的事实来源。不得把持久修正只放在 React 状态、临时 JSON 或日志中。

## 11. 状态流

`parse_status` 使用以下阶段值：

```text
uploaded
→ processing
→ ai_review_pending
→ ai_reviewing
→ human_review_pending
→ correction_pending
→ correction_processing
→ corrected_pending_review
→ indexed_pending_review
```

是否正式可检索继续由 `review_status` 和 `activeIndexVersion` 控制：候选版本保持 `quarantined`，人工发布后才成为 `published`。解析状态和发布状态不得合并成一个字段。

异常状态：

```text
parse_failed
ai_review_failed
correction_failed
reprocess_required
discarded
```

模型服务不可用时，文档进入 `ai_review_failed`，保留规则质检和候选切片。管理员可重试模型预审；如需要降级跳过，必须显式记录人工批准和原因。模型失败不能自动发布。

如果文档已有正式版本，新候选的预审、修正或拒绝不影响当前正式版本。只有发布动作才切换 `activeIndexVersion`。

## 12. 完整“不通过说明”样例

### 12.1 条件和比例被拆散

```text
类型：语义不完整

问题：
当前切片只保留“给付比例为60%”，遗漏前置条件
“未经基本医疗保险结算”，容易被理解为所有住院医疗费用都按60%赔付。

期望修正：
1. 将前置条件和60%比例保留在同一事实切片；
2. 绑定责任“住院医疗保险金”；
3. 不要合并年度免赔额内容；
4. 年度限额保持独立事实块；
5. 召回60%时必须同时返回前置条件；
6. 用问题“未经医保结算时赔付比例是多少”验证。
```

### 12.2 表格脚注遗漏

```text
类型：内容缺失、表格结构错误

问题：
保障计划表已切片，但表格下方“计划三仅适用于指定销售区域”的脚注没有进入任何切片。

期望修正：
框选脚注并重新OCR；脚注独立成限制块，只关联计划三，不能应用到计划一和计划二；
保留表格原图裁剪、表头、计划列和脚注来源坐标。
```

### 12.3 重复页脚混入

```text
类型：内容多余

问题：
第8页切片混入“客服电话95567、扫码关注公众号、第8页”。

期望修正：
从当前切片删除；扫描整份资料底部相同区域，先预览匹配页面，确认后排除重复页脚；
不得删除正文中真实的理赔联系电话。
```

### 12.4 业务图片漏切

```text
类型：图片漏切

问题：
第18页理赔流程图包含“报案、提交材料、审核、给付”四个业务步骤，当前只切了周围标题。

期望修正：
将流程图作为独立业务图片块；保存原图裁剪、OCR文字、流程节点和页码坐标；
与“理赔申请流程”章节建立父子关系，但不能把流程说明解释为必然赔付承诺。
```

### 12.5 其他产品内容串入

```text
类型：产品绑定错误

问题：
第20至23页出现另一产品“康健无忧”的名称和责任表，当前被绑定到“康健长佑”。

期望修正：
重新识别第20至23页产品边界；相关切片不得继续绑定康健长佑；
如果无法确认产品身份，将这些页面隔离并阻止发布，不得简单删除产品名称后继续使用正文。
```

## 13. Contextual Retrieval 与命题化切片的采用方式

### 13.1 Contextual Retrieval

确定性 `contextualPrefix` 默认开启。模型生成背景仅用于召回，并与原文分开存储。模型背景必须说明适用范围，不能引入来源中不存在的合同结论。供应商支持缓存时可以降低重复处理全文的成本，但本设计不依赖某家模型的专有缓存能力。

### 13.2 命题化切片

正式条款不被模型命题改写替代。命题化结果作为 `product_facts` 或检索别名候选保存，并回连原文切片、原图区域和产品版本。只有完整、来源明确且审核通过的事实才能成为确定性回答主路径。

## 14. 错误处理与降级

- OCR 失败：保留原图，允许人工框选重试，不生成无来源文本；
- 图片理解失败：图片块保持隔离，人工可补充分类和说明；
- 模型输出无效 JSON：记录运行失败并重试，不执行部分操作；
- 模型引用不存在的元素或切片：整条建议判为无效；
- 修正操作冲突：阻止生成候选版本，展示冲突对象；
- 批量排除命中正文：影响预览必须显示，人工取消或缩小范围；
- 检索验证失败：保留候选版本和失败问题，返回修正队列；
- 人工拒绝模型判断：保存为 false positive，供后续评估，不自动训练或修改规则。

## 15. 验证与验收

### 15.1 单元测试

- 来源元素覆盖计算；
- 图片分类和装饰图排除；
- 合并、拆分、补入和排除操作；
- `required_context` 关系；
- 产品页面范围重绑；
- 模型 JSON Schema 和来源 ID 校验；
- 候选版本生成、拒绝、发布和回滚。

### 15.2 端到端场景

至少覆盖：

- 条件与比例被拆散；
- 跨页责任和限制；
- 表格脚注遗漏；
- 保障图和流程图漏切；
- 页眉、页脚和二维码批量排除；
- 其他产品内容串入；
- OCR 数字错误；
- 模型误报和模型服务失败；
- 已有正式版本时拒绝新候选不影响线上检索。

### 15.3 质量指标

- 有效来源元素切片覆盖率 ≥ 99%；
- 合同数字与原图/OCR一致率 = 100%；
- 必要条件和限制遗漏率 ≤ 1%；
- 产品和版本绑定准确率 ≥ 99%；
- AI 问题的来源锚点有效率 = 100%；
- 未经人工确认的模型修正进入正式索引次数 = 0；
- 页眉、页脚、二维码进入正式检索的次数 = 0；
- 黄金问题集 Top-5 同时命中主事实和必要限制的比例 ≥ 98%。

外部材料声称的召回提升只作参考。本项目以真实保险黄金问题集、错版本污染率、引用页码准确率和必要限制召回率作为验收依据。

## 16. 分阶段实现

### 第一阶段：可审、可修、可追溯

- 原图与切片对照；
- 稳定来源元素和 `sourceRegions`；
- AI 来源覆盖、语义完整性和关系预审；
- 问题队列、不通过说明和修正计划；
- 编辑、拆分、合并、补入、排除、重新 OCR；
- 审核运行、问题和修正记录入 SQLite；
- 新候选版本和人工发布。

### 第二阶段：图片、表格和检索验证

- 业务图片独立切片；
- 表格原图、行列、脚注和事实关联；
- 自动生成验证问题；
- 主事实与 `required_context` 联合召回检查；
- 模型误报统计。

### 第三阶段：质量优化

- 章节父块替代页面父块；
- 受控 Contextual Retrieval；
- 更多结构化事实和命题候选；
- 基于黄金集优化模型提示、规则和检索，不允许自动学习直接改变正式数据。

## 17. 明确不做

- 第一版不引入新的向量数据库或图数据库；
- 不让模型自由执行 SQL 或任意修正动作；
- 不把所有图片都送入 RAG；
- 不将通用固定 overlap 作为主要解决方案；
- 不因模型预审通过而自动发布；
- 不把模型生成说明当作合同原文；
- 不自动把单份资料的排除规则扩散到其他资料。
