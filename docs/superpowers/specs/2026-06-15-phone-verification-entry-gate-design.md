# 手机验证码录单门槛 — 设计文档

> 日期: 2026-06-15
> 状态: 用户已确认设计
> 范围: 客户端保险责任查询免登录，保单照片上传和保单保存必须先完成手机号验证码登录

## 目标

调整客户录单入口的手机号验证规则。

“输入保险名称查保险责任”是公开查询能力，不要求客户用手机号验证码登录。客户只输入保险公司和保险名称时，可以直接查询本地匹配和公开保险责任。

“录入保单”是账户数据写入和客户保单 OCR 流程，必须先完成手机号验证码登录。客户用保单照片识别保单信息前需要验证手机号；客户保存保单前也需要验证手机号。

第一期不新增新的登录方式，不改会员付费规则，不改责任查询的数据来源，只把现有入口的鉴权边界收紧并统一到后端。

## 已确认决策

- 右下角“输入保险名称查责任”继续免手机号验证码。
- 录单页当前没有独立的“生成保险责任”按钮，因此不新增该按钮。
- 录单页现有两个用户动作需要手机号验证码:
  - 上传保单照片；
  - 保存保单。
- 后端仍保护旧接口或隐藏入口，`/api/policies/analyze` 和其它录单 API 一样要求已登录。
- 废弃“游客第一张保单可直接录入，第二张开始需要验证手机号”的规则。
- 会员额度仍只在已登录用户保存保单时判断，和本次手机号门槛分开。

## 范围外

- 不做微信一键登录、小程序登录、密码登录或第三方登录。
- 不改变短信发送、验证码校验、会话 token 的底层实现。
- 不改变保险责任查询算法、知识库匹配规则、OCR 识别逻辑。
- 不改变会员价格、免费额度、支付和订单逻辑。
- 不调整本机生产环境配置、真实短信配置或 `.env.local`。

## 当前上下文

客户前端在 `src/apps/customer/CustomerApp.tsx` 里已有手机号验证弹窗和 token 状态。验证码接口在 `server/routes/auth.routes.mjs`，验证成功后返回 bearer token。

保险责任助手走 `src/api/contracts/responsibility.ts`，后端入口是 `/api/policy-responsibilities/*`。这些接口当前没有依赖 `resolveAuthUser`，适合作为公开查询入口保留。

录单 OCR 和保存走 `src/api/contracts/policy.ts`:

- `recognizePolicy` -> `/api/policies/recognize`
- `analyzePolicy` -> `/api/policies/analyze`
- `scanPolicy` -> `/api/policies/scan`

后端当前使用 `assertGuestCanScan` 实现“游客第一张可录、第二张验证”的规则。新规则应把这类入口改为“未登录不可录单”，而不是继续计算游客已保存数量。

## 推荐方案

采用“后端统一门槛，前端提前拦截”的方案。

后端是最终安全边界。所有 `/api/policies/*` 中会触发 OCR、分析或保存保单的录单入口都要求已登录用户。没有有效 token 时返回 `REGISTRATION_REQUIRED`，前端据此弹出手机号验证码。

前端负责把体验做顺。未登录用户点上传照片时先弹验证码，不打开文件选择器；点保存保单时先弹验证码，不进入保存请求。这样用户不会先选完照片或填完保存动作才被后端拦住。

没有选择“只改前端拦截”，因为 API 直调仍可绕过。没有选择“新增一次性录单验证码状态”，因为现有手机号登录 token 已能表达用户身份，额外状态会增加保存、迁移和会员额度判断复杂度。

## 架构

### 公开查询入口

保持以下路由不要求登录:

- `POST /api/policy-responsibilities/query`
- `POST /api/policy-responsibilities/matches`
- `GET /api/policy-responsibilities/company-suggestions`
- `GET /api/policy-responsibilities/product-suggestions`

这些接口只返回公开产品责任和知识库匹配，不保存客户保单，不读取客户名下保单，不写入客户账户数据。

### 录单入口

以下路由必须要求登录:

- `POST /api/policies/recognize`
- `POST /api/policies/analyze`
- `POST /api/policies/scan`

后端先解析 bearer token。如果没有登录用户，立即返回:

```json
{
  "ok": false,
  "code": "REGISTRATION_REQUIRED",
  "message": "录入或上传保单前需要先完成手机验证码",
  "registrationRequiredNext": true
}
```

已登录用户继续走现有 OCR、分析、保存、家庭绑定和会员额度逻辑。

### 权益校验

手机号验证只解决“谁在录入”。会员额度只解决“这个已登录用户还能不能继续保存”。

保存入口的顺序应为:

1. 验证用户已登录。
2. 保存前检查会员额度。
3. 解析 scan 或 OCR 输入。
4. 构造家庭绑定和保单。
5. 持久化保单。

上传识别入口只要求登录，不扣会员额度。

## 前端交互

未登录用户仍能进入应用并打开责任助手。

未登录用户点右下角“输入保险名称查责任”:

- 不弹验证码。
- 可以选择保险公司、保险产品。
- 可以查询责任明细。
- 查询失败只显示责任查询本身的错误。

未登录用户点“上传保单照片”:

- 先弹手机号验证码。
- 不打开文件选择器。
- 验证成功后关闭弹窗，用户再次点击上传照片继续。

未登录用户点“保存保单”:

- 先弹手机号验证码。
- 不发保存请求。
- 验证成功后保留当前表单内容，用户再次点击保存继续。

如果前端状态过期或用户通过旧页面绕过前端拦截，后端返回 `REGISTRATION_REQUIRED` 时继续复用现有错误处理，弹手机号验证码。

验证弹窗文案改为:

> 录入或上传保单前需要验证手机号；仅查询保险责任无需验证。

删除或替换所有“第一张保单可直接录入，第二张开始需要验证手机号”的客户可见文案。

## 数据流

### 公开责任查询

1. 用户打开责任助手。
2. 前端调用 `/api/policy-responsibilities/matches` 或 suggestion 接口。
3. 用户选择产品后调用 `/api/policy-responsibilities/query`。
4. 后端使用本地知识库和责任分析返回结果。
5. 不创建用户、不创建 session、不写入保单。

### 上传保单照片

1. 用户点击上传。
2. 前端检查本地 token。
3. 没有 token 时打开手机号验证弹窗。
4. 用户完成验证码后，前端保存 token。
5. 用户再次点击上传并选择照片。
6. 前端调用 `/api/policies/recognize`。
7. 后端确认 token 对应用户，再执行 OCR 和本地责任草稿匹配。

### 保存保单

1. 用户点击保存。
2. 前端检查本地 token。
3. 没有 token 时打开手机号验证弹窗并保留表单。
4. 用户完成验证码后，前端保存 token。
5. 用户再次点击保存。
6. 后端确认 token 对应用户。
7. 后端检查会员额度。
8. 后端保存保单、家庭绑定、责任和现金流相关数据。

## 错误处理

### 未登录

后端统一返回 `REGISTRATION_REQUIRED`。前端统一打开手机号验证弹窗。

推荐错误文案:

- 上传照片: `上传保单照片前需要先验证手机号`
- 保存保单: `保存保单前需要先验证手机号`
- API 兜底: `录入或上传保单前需要先完成手机验证码`

### token 失效

沿用现有 `clearCustomerSession` 行为，清理 token 并提示重新验证手机号。用户当前表单内容不应因为 token 失效被清空。

### 会员额度不足

仍返回 `MEMBERSHIP_REQUIRED`，前端打开会员购买弹窗。这个错误只可能发生在已登录保存保单时，不应出现在公开责任查询或上传识别阶段。

## 测试计划

### 后端测试

- 未登录直调 `/api/policies/recognize` 返回 `401 REGISTRATION_REQUIRED`。
- 未登录直调 `/api/policies/analyze` 返回 `401 REGISTRATION_REQUIRED`。
- 未登录直调 `/api/policies/scan` 返回 `401 REGISTRATION_REQUIRED`，且不新增保单。
- 登录后 `/api/policies/recognize` 可以识别。
- 登录后 `/api/policies/scan` 可以保存。
- 公开责任查询接口未登录仍可调用。
- 已登录保存超过会员免费额度仍返回 `MEMBERSHIP_REQUIRED`，证明会员逻辑保留。

更新 `tests/customer-ui-style.test.mjs` 或新增聚焦 UI 测试，覆盖:

- 保存入口在未登录时会先打开手机号验证弹窗。

## 验收标准

- 未登录用户可以使用右下角责任助手查询保险责任。
- 未登录用户点击上传保单照片时先看到手机号验证码弹窗，不打开文件选择器。
- 未登录用户点击保存保单时先看到手机号验证码弹窗，不写入保单。
- 未登录用户直调 `/api/policies/recognize`、`/api/policies/analyze`、`/api/policies/scan` 均被后端拒绝。
- 已登录用户上传照片和保存保单正常。
- 会员免费额度和购买提示只在已登录保存保单时生效。
- 客户可见文案不再表达“第一张可直接录入，第二张才验证”。

## 迁移与兼容

已有游客保单数据不在本次设计中删除或迁移。新规则上线后，未登录游客不能再新增识别或保存。已有手机号登录用户和会员数据不受影响。

旧前端或缓存页面如果仍尝试游客录单，会被后端 `REGISTRATION_REQUIRED` 拦截。前端应保持现有错误兜底，提示用户验证手机号。
