# 家庭档案报告统计与分享 — 设计文档

> 日期: 2026-06-01
> 状态: 用户已确认设计
> 范围: 家庭档案、家庭成员、保单录入归属、旧数据迁移、家庭报告统计、分享和导出

## 目标

把家庭报告的统计边界从“按保单中的被保人姓名临时分组”升级为“按家庭档案和家庭成员 ID 统计”。录入保单时先选择家庭档案，投保人和被保险人必须来自同一个家庭档案。家庭报告、分享链接和导出都只展示当前家庭档案内的数据。

第一版优先保证录入链路改动少、旧数据不丢失、报告统计边界清楚。

## 已确认决策

- 业务对象使用“家庭档案”，不是“客户档案”。
- 系统可以管理不同家庭档案；一张保单只能归属一个家庭档案。
- 新建家庭档案只输入家庭名称。
- 家庭档案有一个家庭关系中心，后端字段为 `coreMemberId`。
- 核心人员不在新建家庭档案时强制设置，而是在首次录入保单、OCR 识别后顺手设置。
- 投保人和被保险人都必须是当前家庭档案下的成员。
- 家庭成员可以在录入保单时新增。
- 家庭报告按 `familyId` 取数，按 `insuredMemberId` 做成员统计，投保人用于展示保单关系和缴费视角。
- 现有数据库先自动迁移到默认家庭档案，核心人员临时自动选择，关系后续允许用户修正。

## 推荐方案

采用“家庭档案 + 家庭成员表 + 保单绑定成员 ID”的方案。

没有采用继续按姓名分组的方案，因为姓名字符串容易受 OCR 错字、同名、改名影响，无法稳定支持家庭身份、分享边界和后续数据清理。没有采用“成员表存在但保单仍只存姓名”的折中方案，因为统计仍会依赖模糊匹配，后续维护成本高。

## 数据模型

### FamilyProfile

家庭档案是报告和录入的归属边界。

```ts
type FamilyProfile = {
  id: number;
  ownerUserId?: number | string;
  familyName: string;
  coreMemberId: number | null;
  status: 'active' | 'archived';
  createdAt: string;
  updatedAt: string;
};
```

### FamilyMember

家庭成员只属于一个家庭档案。家庭身份统一表达为“与核心人员的关系”。

```ts
type FamilyMember = {
  id: number;
  familyId: number;
  name: string;
  relationToCore: 'self' | 'spouse' | 'son' | 'daughter' | 'father' | 'mother' | 'parent' | 'parent_in_law' | 'grandparent' | 'sibling' | 'other' | 'pending';
  relationLabel: string;
  role: 'core' | 'adult' | 'child' | 'elder' | 'unknown';
  gender?: 'male' | 'female' | 'unknown';
  birthday?: string;
  idNumberTail?: string;
  mobile?: string;
  notes?: string;
  status: 'active' | 'archived';
  createdAt: string;
  updatedAt: string;
};
```

### Policy 扩展字段

保单保留当前 OCR 文本字段，同时新增稳定关联字段。

```ts
type PolicyFamilyFields = {
  familyId: number;
  applicantMemberId: number;
  insuredMemberId: number;
  applicantNameSnapshot: string;
  insuredNameSnapshot: string;
  applicantRelationSnapshot: string;
  insuredRelationSnapshot: string;
  participantReviewStatus?: 'ok' | 'name_mismatch' | 'pending_member';
};
```

兼容规则:

- 新统计优先使用 `familyId`、`applicantMemberId`、`insuredMemberId`。
- 现有 `applicant`、`insured`、`applicantRelation`、`insuredRelation` 保留为 OCR 展示、历史兼容和人工核对字段。
- 如果旧保单尚未完成成员绑定，报告可以显示在迁移待整理列表中，但不再作为新录入口径。

## 新建家庭档案

新建家庭档案只要求输入:

- 家庭名称

不要求先填写核心人员、家庭成员、生日、证件等信息。这样用户可以先建档，再通过拍照 OCR 的真实保单信息补齐人员。

## 保单录入流程

1. 先选择家庭档案
   - 如果只有一个家庭档案，可以默认选中。
   - 如果没有家庭档案，先弹出轻量创建框，只输入家庭名称。

2. 拍照或上传保单
   - OCR 识别投保人、被保险人、生日、证件号、保单号等字段。
   - OCR 结果先进入核对状态，不直接进入报告统计。

3. 如果家庭没有核心人员，在当前录入页设置
   - 不跳转到家庭档案详情页。
   - 文案使用“家庭关系中心”或“以谁作为家庭关系基准”，后台仍保存为 `coreMemberId`。
   - 候选项优先来自本次 OCR:
     - `张三｜投保人`
     - `李四｜被保险人`
     - 从已有家庭成员中选择
     - 新增人员
   - 选中后该成员设为 `relationToCore = self`、`role = core`。
   - 其他成员关系可先标记为 `pending` 或 `other`，不强制当场补全。

4. 匹配或新增投保人、被保险人
   - 系统在当前家庭成员中按姓名、生日、证件后四位匹配。
   - 精确匹配时自动选中。
   - 姓名相同但生日或证件冲突时要求人工确认。
   - 找不到时显示“新增为家庭成员”，最少填写姓名和与核心人员关系。

5. 保存前校验
   - 必须已选择 `familyId`。
   - 必须已设置 `coreMemberId`。
   - 必须已选择或新增 `applicantMemberId`。
   - 必须已选择或新增 `insuredMemberId`。
   - 投保人和被保险人必须都属于当前 `familyId`。
   - OCR 姓名和选中成员姓名不同可以保存，但标记为 `name_mismatch`，在报告和待整理项中提示。

6. 保存保单
   - 写入 `familyId`、`applicantMemberId`、`insuredMemberId`。
   - 写入姓名和关系快照，保证历史报告和 OCR 核对可追溯。

## 家庭档案页面

### 家庭档案列表

列表用于选择和管理不同家庭档案。字段:

- 家庭名称
- 核心人员
- 成员数
- 保单数
- 年交保费
- 最近更新时间
- 操作: 查看报告、编辑家庭、录入保单、分享报告

### 家庭档案详情

详情页维护成员和待整理项。字段:

- 姓名
- 家庭身份
- 与核心人员关系
- 生日
- 手机号或证件后四位
- 关联保单数
- 状态

支持:

- 新增成员
- 编辑成员
- 归档成员
- 设置或更换核心人员
- 处理待整理项

更换核心人员时，其他成员的 `relationToCore` 统一进入待确认状态，要求用户重新确认关系。

## 家庭报告统计

报告入口先选择家庭档案。统计规则:

- 只读取 `policy.familyId = 当前家庭档案 id` 的保单。
- 家庭成员数取当前家庭下 active 成员数。
- 被保人维度按 `insuredMemberId` 分组。
- 投保人维度用于保单清单展示、投保关系展示和保费承担视角。
- 成员标题展示为 `姓名｜家庭身份｜被保人`，例如 `王小明｜儿子｜被保人`。
- 现有重疾、意外、财富、雷达图结构继续沿用，但底层分组从姓名改为成员 ID。

保单清单新增展示字段:

- 投保人
- 被保人
- 被保人家庭身份
- 投保人与被保人关系
- 年交保费
- 保障/保额
- 现金价值
- 数据状态
- 姓名不一致提示

## 分享和导出

- 分享的是某一个家庭档案的报告，不分享全库。
- 分享链接绑定 `familyId` 和报告快照时间。
- 分享页只读，不能编辑成员或保单。
- 分享快照保留成员展示名称、家庭身份、保单快照字段，避免后续改名导致历史报告变化。
- PDF 和长图导出按当前家庭档案生成，字段顺序与报告页一致。

## 旧数据迁移

迁移目标是让现有数据先可用，不在迁移阶段强制用户补齐全部家庭关系。

迁移步骤:

1. 为现有保单创建一个默认家庭档案。
2. 从旧保单的 `applicant`、`insured` 中提取人员姓名。
3. 按姓名去重生成 `FamilyMember`。
4. 核心人员自动选择:
   - 优先选出现次数最多的人。
   - 如果无法判断，选第一个识别到的人。
5. 自动核心人员设置为 `relationToCore = self`、`role = core`。
6. 其他成员设置为 `relationToCore = pending`、`relationLabel = 待确认`。
7. 旧保单按投保人和被保险人姓名匹配成员 ID。
8. 无法匹配的保单保留在默认家庭档案，标记为 `pending_member`，进入待整理项。

## 错误处理和边界规则

- 没有家庭档案: 录入页弹出轻量创建框，只输入家庭名称。
- 没有核心人员: 不跳转页面，在录入页通过 OCR 候选或已有成员设置家庭关系中心。
- 没有家庭成员: 可以从 OCR 的投保人和被保险人新增。
- 投保人或被保险人不属于当前家庭: 禁止保存保单。
- OCR 姓名和选中成员不同: 允许保存，标记姓名不一致待核对。
- 成员被归档: 历史保单继续统计；新保单默认不允许选择归档成员，除非用户显式展开。
- 删除家庭档案: 第一版只归档，不硬删除。
- 分享报告: 只读，不允许编辑。

## 测试要求

### 迁移测试

- 旧保单能生成默认家庭档案。
- 投保人和被保险人能生成去重后的家庭成员。
- 核心人员能按出现次数或首个识别人员自动选择。
- 旧保单能写入 `familyId`、`applicantMemberId`、`insuredMemberId`。
- 无法匹配成员的保单进入待整理项。

### 录入测试

- 录入页必须先有家庭档案。
- 没有家庭档案时能轻量创建。
- 没有核心人员时能在录入页从 OCR 候选中设置。
- 投保人和被保险人只能选择当前家庭成员。
- 未匹配 OCR 人员能新增为家庭成员。
- 跨家庭成员不能保存到同一张保单。

### 报告测试

- 家庭报告只统计当前 `familyId` 的保单。
- 成员分组按 `insuredMemberId`，不是姓名字符串。
- 保单清单展示投保人、被保人和家庭身份。
- 姓名不一致的保单显示核对提示。
- 现有重疾、意外、财富、雷达图仍能按成员展示。

### 分享测试

- 分享链接只包含目标家庭档案数据。
- 分享页不泄漏其他家庭档案的成员和保单。
- 分享页只读。
- 导出 PDF 或长图使用同一个家庭档案统计口径。

## 非目标

- 第一版不做一个自然人跨多个家庭档案复用。
- 第一版不做家庭成员关系图谱，只维护成员与核心人员的关系。
- 第一版不做家庭档案硬删除。
- 第一版不做行业标准保额建议或产品推荐。
- 第一版不把分享页做成可编辑协作页面。
